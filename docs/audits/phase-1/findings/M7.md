## Scope

Area: `crates/forge-session`. Threat class: **T1 — tool-approval fidelity**. The four-scope approval UI promises Once / ThisFile / ThisPattern / ThisTool semantics. When the daemon collapses all four into `Once`, the user experience silently degrades — "Approve for this tool" still re-prompts every call, leading to approval fatigue and click-through.

## Finding

- **Location:** `crates/forge-session/src/server.rs:337-342`, `crates/forge-session/src/orchestrator.rs:176`, `:222`
- **Severity:** medium
- **Threat class:** T1

The IPC `ToolCallApproved` struct has a `scope: String` field. `server.rs:337-342` reads only `a.id` — `a.scope` is dropped. The orchestrator unconditionally emits `Event::ToolCallApproved { scope: ApprovalScope::Once, ... }`.

Consequences:
1. The UI's "Approve for this tool" / "this pattern" buttons behave identically to "Once" — every subsequent similar tool call re-prompts.
2. If a future implementation honors wider scopes, the lossy path continues emitting `Once`, masking a downstream wider-scope bug.

Not a bypass — nothing executes without approval — but a consent-fidelity gap that trains users to click-through.

## Reproduction

1. Send `ToolCallApproved { scope: "ThisTool" }`.
2. Observe the emitted `ToolCallApproved` event contains `scope: Once`.
3. Trigger a second tool call of the same tool — the approval gate fires again.

## Remediation

Parse `a.scope` into `forge_core::ApprovalScope` at `server.rs:337`. Expand the oneshot channel to carry `(bool, ApprovalScope)` (or a typed enum). Pass the resulting scope into the `ToolCallApproved` event in `orchestrator.rs`.

Decide separately whether wider scopes should short-circuit future approval prompts (design question for Phase 2). At minimum, record the scope faithfully in the event log today.
