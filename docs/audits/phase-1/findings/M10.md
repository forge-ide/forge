## Scope

Area: `crates/forge-shell`. Threat class: **T5 — webview isolation**. Trust between sessions is currently enforced by a one-line client-side check — exactly the wrong place for a boundary. Under any XSS gadget (H9 enables this), one session's webview reads every other session's event stream.

## Finding

- **Location:** `crates/forge-shell/src/ipc.rs:36-41` (`AppHandleSink::emit`)
- **Severity:** medium
- **Threat class:** T5 / T6

`AppHandleSink::emit` calls `AppHandle::emit("session:event", payload)`, which broadcasts to every window in the app (dashboard + every `session-*` window). Filtering happens in the Solid store (`web/packages/app/src/routes/Session/SessionWindow.tsx:47` does `if (payload.session_id !== id) return;`).

Consequences:
1. A Session-B window's JS can subscribe to `session:event` and observe Session A's full event stream (tool-call args, assistant turns, content excerpts).
2. Performance cost scales linearly in window count.
3. The trust boundary between sessions is enforced client-side.

## Reproduction

1. Open Session A, send a message, start a tool call.
2. In any other window's devtools: `import('@tauri-apps/api/event').then(m => m.listen('session:event', e => console.log(e.payload)))`.
3. Observe payloads for Session A.

## Remediation

Use `AppHandle::emit_to` with a targeted webview:

```rust
app.emit_to(
    EventTarget::webview_window(format!("session-{}", session_id)),
    "session:event",
    payload,
)
```

Single-file change in `src/ipc.rs`. Keep the client-side `session_id` check as defense-in-depth.
