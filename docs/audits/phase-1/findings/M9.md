## Scope

Area: `crates/forge-session`. Threat class: **T4 — sandbox completeness**. Level 1 isolation's stated guarantee is that a misbehaving or malicious tool call cannot take down the host. CPU and AS limits are per-process; children inherit their own budget, so a fork-bomb or fd-exhaustion is within reach of a single approved tool call.

## Finding

- **Location:** `crates/forge-session/src/sandbox.rs:26-33` (`SandboxConfig`), `:149-163` (`pre_exec`)
- **Severity:** medium
- **Threat class:** T4

Only `RLIMIT_CPU` (30s) and `RLIMIT_AS` (512 MiB) are applied. These are per-process: a shell that forks children has each child getting its own 30s CPU budget.

- `RLIMIT_NPROC` not set → a single approved `shell.exec` can fork-bomb the calling uid.
- `RLIMIT_NOFILE` not set → fd-exhaustion is possible.
- `RLIMIT_FSIZE` not set → cat-to-disk can fill the filesystem.

All are post-approval effects, but a single approved innocuous-looking tool call can cascade.

## Reproduction

Approve a `shell.exec` that forks a rapidly-replicating process. Observe uid process count climbs to host-default `RLIMIT_NPROC`; no per-sandbox cap is enforced.

## Remediation

Extend `SandboxConfig` with additional defaults:
- `RLIMIT_NPROC ≈ 64`
- `RLIMIT_NOFILE ≈ 256`
- `RLIMIT_FSIZE ≈ 100 MiB`

Apply in `pre_exec` alongside existing `apply_rlimit` calls.

Longer-term: move to cgroup-based limits so they apply to the pgid as a whole, not per-process.
