## Scope

Area: `crates/forge-shell`. Threat class: **T7 — deserialization / type confusion**. Shell is the boundary that should normalize attacker-controllable shapes. A crafted scope that passes a future `starts_with`/`contains` check could widen approval silently.

## Finding

- **Location:** `crates/forge-shell/src/ipc.rs:85-97`, `crates/forge-ipc/src/lib.rs:33-38`
- **Severity:** low
- **Threat class:** T7

On the wire, approval scope is a free-form `String` (`ToolCallApproved { id, scope: String }`) matched against `"Once" | "ThisFile" | "ThisPattern" | "ThisTool"` later in `forge-session`. The shell does no validation; the webview can pass any string. A non-matching string is silently treated as "no persistent approval" by any `match` with a catch-all arm.

A crafted scope that happens to satisfy a future `starts_with` or `contains` check could widen approval beyond what the UI rendered.

Note: this finding is closely related to M7 (scope is currently ignored by the server). Both should be fixed together — parse to typed enum at the shell boundary *and* forward it properly in the session daemon.

## Reproduction

```js
await invoke('session_approve_tool', { sessionId: id, toolCallId: 'abc', scope: 'ThisTool-but-also-more' })
// Shell forwards verbatim
```

## Remediation

Change the shell-side DTO to a serde-enum:

```rust
enum ApprovalScope { Once, ThisFile, ThisPattern, ThisTool }
```

Matches the TS-side `ApprovalScope` in `@forge/ipc`. Fail the command when serde cannot deserialize.
