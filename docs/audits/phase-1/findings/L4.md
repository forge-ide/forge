## Scope

Area: `crates/forge-shell`. Threat class: **T7 — command deserialization / resource exhaustion**. The 4 MiB `forge-ipc` wire cap prevents an OOM crash but not the cost/abuse vector — repeated 4 MiB sends from a compromised webview produce transient heap pressure and real billing against the user.

## Finding

- **Location:** `crates/forge-shell/src/ipc.rs:72-83`, `crates/forge-shell/src/bridge.rs:193-198`
- **Severity:** low
- **Threat class:** T7

`session_send_message(session_id, text: String, ...)` accepts arbitrarily large `text` and forwards it as `SendUserMessage`. `forge_ipc::write_frame` rejects frames above 4 MiB, so larger payloads fail late after serde has already allocated. A loop of 4 MiB sends produces 4 MiB per call of transient Rust-side allocation plus a full frame on the UDS, reaching the model provider and billing against the user (or blowing context).

Same concern for `tool_call_id` and `scope`/`reason` strings.

## Reproduction

```js
await invoke('session_send_message', { sessionId: id, text: 'A'.repeat(4_000_000) })
// Repeat in a tight loop
```

## Remediation

Cap `text` at a reasonable chat-turn size (e.g. 128 KiB) in `session_send_message` and return a typed error above. Same for `tool_call_id` (short, ~64 chars) and `scope`/`reason`.
