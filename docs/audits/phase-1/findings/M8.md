## Scope

Area: `crates/forge-session`. Threat class: **T1 (approval fidelity) + T3 (shell path confusion)**. The approval preview is the user's sole check on what a command will do. When an attacker-chosen `cwd` does not appear in the preview, the user consents to an innocuous-looking command that runs in a directory context they never saw.

## Finding

- **Location:** `crates/forge-session/src/tools/shell_exec.rs:35-54` (preview), `:88-95` (cwd resolution)
- **Severity:** medium
- **Threat class:** T1 / T3

The `cwd` argument on `shell.exec` is taken verbatim from tool args with no constraint that it sits inside `ctx.workspace_root`. The approval preview composes `Run command: {command} {args}` — **cwd is omitted**.

A model can issue `{command: "ls", args: ["."], cwd: "/etc"}` and the user clicks approve on `Run command: ls .`. Combined with missing `RLIMIT_FSIZE` and a `shell.exec` that writes, e.g. `{command: "/bin/sh", args: ["-c", "cat /etc/passwd > /tmp/x"], cwd: "/tmp"}`, the user has no visible consent signal for the directory context.

## Reproduction

Approve a `shell.exec` whose args mention only an innocuous command; inspect the tool result to see it executed against a cwd the user never saw.

## Remediation

1. Include `cwd` in `ApprovalPreview.description` when it differs from `workspace_root`.
2. Validate `cwd` canonicalizes under `ctx.workspace_root` or reject the call (symmetric with fs.* symlink/root checks in `forge-fs`).
