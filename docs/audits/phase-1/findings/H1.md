## Scope

Area: `crates/forge-cli`. Threat class: **T12 — raw-PID SIGTERM**. The CLI ships an unsafe signal primitive whose input comes from a file without any sign/liveness check. POSIX `kill(2)` specifies that pid 0 signals the caller's process group and pid -1 signals every process the user may signal — a "kill the daemon" command that falls into either case detonates outside the intended target.

## Finding

- **Location:** `crates/forge-cli/src/main.rs:194`
- **Severity:** high
- **Threat class:** T12a (raw-PID signal input validation)

`session_kill` parses the pid file with `raw.trim().parse::<libc::pid_t>()` and passes the result directly to `libc::kill(pid, SIGTERM)` with no sign check. POSIX `kill(2)`:
- `pid == 0` → "signal every process in the caller's process group"
- `pid == -1` → "signal every process the user may signal"
- `pid < -1` → "signal process group `abs(pid)`"

Two reachable paths:
1. `run_agent` (`main.rs:239`) writes `child.id().unwrap_or(0)` — if the `forged` child is already reaped/failed, the pid file literally contains `0`.
2. Any same-uid process (or an attacker with write access to `$XDG_RUNTIME_DIR/forge/sessions/`) can plant a matching `<id>.sock` + `<id>.pid` where the pid file contents are `0` or `-1`. `session list` enumerates it, the user pastes the id, `session kill <id>` then broadcasts SIGTERM across their login session.

## Reproduction

1. `echo -n 0 > $XDG_RUNTIME_DIR/forge/sessions/deadbeef.pid` and touch the matching `.sock`.
2. `forge session kill deadbeef`.
3. `libc::kill(0, SIGTERM)` fires, terminating every process in the forge CLI's process group.

Alternatively: trigger `run_agent` against a `forged` binary that exits before `child.id()` is read, so the pid file gets `"0"`.

## Remediation

Immediately before the unsafe block at `main.rs:194`:

```rust
anyhow::ensure!(pid > 0, "refusing to signal non-positive pid {pid}");
```

Also fix the upstream bug at `main.rs:239` — do not write a pid file when `child.id()` returns `None`; return an error or skip pid-file creation. Consider parsing as `u32` then converting, which naturally rejects negatives.
