## Scope

Area: `crates/forge-fs`. Threat class: **T2 â€” path traversal / `allowed_paths` bypass** (with confidentiality side-channel). The four-scope approval UI depends on users seeing the right information *before* they click approve. When the approval preview itself discloses attacker-chosen file contents, the consent control becomes a confidentiality leak *regardless* of whether the user approves.

## Finding

- **Location:** `crates/forge-fs/src/mutate.rs:111-122` (`write_preview`)
- **Also:** `crates/forge-session/src/tools/fs_write.rs:15-22`, `crates/forge-session/src/orchestrator.rs:187-192`
- **Severity:** high
- **Threat class:** T2

`forge_fs::write_preview(path: &str, content: &str)` calls `std::fs::read_to_string(path)` directly â€” no canonicalization, no symlink check, no `allowed_paths` parameter on the function signature. `FsWriteTool::approval_preview` is called from the orchestrator in manual-approval mode and passes the LLM-supplied path straight through. The resulting unified diff (every line of the target file prefixed with `-`) is embedded into `ApprovalPreview.description` and emitted on the session event bus as `Event::ToolCallApprovalRequested`. The leak occurs **before** the user sees the approval dialog â€” rejecting the tool call does not un-leak content already delivered to the UI event stream.

This bypasses every boundary the rest of the crate carefully enforces (canonicalize, symlink rejection, `allowed_paths` glob). Only triggers when `auto_approve` is false, but that is the default secure mode.

## Reproduction

1. Start a session with `allowed_paths` restricted to `['/home/user/work/**']`, `auto_approve = false`.
2. Have the agent emit `fs.write` with args `{"path": "/etc/passwd", "content": ""}`.
3. Orchestrator calls `FsWriteTool::approval_preview` â†’ `forge_fs::write_preview("/etc/passwd", "")` â†’ `read_to_string("/etc/passwd")`.
4. `ApprovalPreview.description` carries the full contents of `/etc/passwd` on the event bus.
5. Same vector works for `~/.ssh/id_rsa`, `~/.aws/credentials`, `.env`, etc.

## Remediation

**Preferred:** remove the "diff against existing" behavior from `write_preview` entirely â€” emit a content-only preview (proposed new content + byte count). Any "existing vs proposed" diff should be computed inside `invoke()` *after* enforcement, or via an explicit `fs.read` call that goes through the same approval gate.

**Minimal:** change `write_preview`'s signature to accept `allowed_paths: &[String]` and run the same `canonicalize_no_symlink + enforce_allowed` gauntlet as `write`/`edit` before reading. On denial, return an `ApprovalPreview` whose description reveals the path and reason, never on-disk content.
