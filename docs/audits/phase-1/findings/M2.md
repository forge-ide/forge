## Scope

Area: `crates/forge-core`. Threat class: **T7 — IPC deserialization / size limits**. Not a direct remote-attacker primitive today (the daemon writes the event log), but `assistant_delta` and `user_message` variants carry model-derived / user-derived strings. An adversarial provider/proxy can inflate them; a rogue `fs.write` tool call lands a crafted file; either way, readers later OOM.

## Finding

- **Location:** `crates/forge-core/src/event_log.rs:111-133` (`read_since`), `crates/forge-core/src/transcript.rs:34-44` (`Transcript::from_file`)
- **Severity:** medium
- **Threat class:** T7

Both readers iterate `events.jsonl` with `BufReader::lines()` / `Lines::next_line()` and pass each `String` to `serde_json::from_str`. No per-line byte cap. A malformed / adversarial `events.jsonl` with a single unterminated line of N bytes is buffered in full (O(N) heap) before parsing.

`Event::ToolCallStarted { args: Value, ... }` and `ToolCallCompleted { result: Value, ... }` also have no depth bound on the resulting `serde_json::Value`.

The schema-header check only guards the first line; line 2+ is unbounded.

## Reproduction

Craft an `events.jsonl` with one very large line (2 GB of content inside a single JSON-object line). Call `forge_core::read_since(path, 0).await` — RSS grows ~2 GB before `from_str` runs.

## Remediation

Use `take` / a custom `AsyncBufRead` wrapper to cap per-line reads at a documented ceiling (4 MiB is conservative — current legitimate events are much smaller because the orchestrator chunks assistant deltas).

```rust
buf_reader.take(MAX_LINE).read_to_end(...)
```

When the limit exhausts without a newline, return `ForgeError::Other(anyhow!("events.jsonl line at seq {seq} exceeds {MAX_LINE} bytes"))`. Apply symmetrically in `Transcript::from_file`.

Consider adding a depth cap on `args: Value` / `result: Value` via a visitor — `serde_json::from_str` does not cap depth by default.
