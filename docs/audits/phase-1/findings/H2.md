## Scope

Area: `crates/forge-cli`. Threat class: **T12 — raw-PID SIGTERM**. Even when the pid is positive, a race against kernel PID reuse means `SIGTERM` can hit an unrelated process — a text editor, a build, a browser renderer — that inherited the pid after the original `forged` exited.

## Finding

- **Location:** `crates/forge-cli/src/main.rs:194`
- **Severity:** high
- **Threat class:** T12b (raw-PID reuse / signal-to-wrong-process)

The `SAFETY` comment on line 193 claims "pid comes from a file we wrote", but nothing verifies:
1. The process with that pid is still the `forged` child originally spawned
2. The pid has not been reused by the kernel
3. The process belongs to the current uid

Between `session new` (which writes the pid file at `main.rs:65` via `cmd.spawn()` + `mem::forget(child)`) and `session kill`, the `forged` process may exit and Linux may recycle its pid for any process the user can create. `session_kill` then SIGTERMs that unrelated process. The child handle is explicitly leaked at `main.rs:63` (`std::mem::forget(child)`), so there is no `waitpid`/zombie check possible from the CLI — every invocation is racing against pid reuse.

## Reproduction

1. `forge session new agent foo` — `forged` is spawned, pid file written with PID P.
2. `forged` crashes or exits on its own (the CLI does not clean up pid files when the daemon self-terminates).
3. Later, the kernel assigns PID P to the user's text editor.
4. User runs `forge session kill <id>` — SIGTERM is delivered to the editor.

## Remediation

Preferred: use `pidfd_open(pid, PIDFD_NONBLOCK)` and `pidfd_send_signal` (Linux 5.1+), which is race-free against PID reuse.

Minimal: record `forged`'s start-time (read from `/proc/<pid>/stat` field 22) in the pid file and cross-check at kill time. If start-time differs, the pid has been reused — refuse.

Additionally, `session new` should write the pid file atomically with `O_EXCL`, and `forged` itself should own the pid-file lifecycle (remove on exit) rather than the CLI leaking the child handle.
