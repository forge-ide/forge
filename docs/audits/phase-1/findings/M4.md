## Scope

Area: `crates/forge-providers`. Threat class: **T7 — HTTP / JSON deserialization DoS**. Same class as H4 (streaming) but on the buffered path; lower severity because `list_models()` is called on dashboard refresh, not per chat turn.

## Finding

- **Location:** `crates/forge-providers/src/ollama.rs:49-52` (`resp.json::<serde_json::Value>().await`)
- **Severity:** medium
- **Threat class:** T7

After `is_success()`, `list_models()` calls `resp.json()`, which internally calls `resp.bytes()` and buffers the whole body before `serde_json::from_slice`. No `Content-Length` enforcement, no reqwest-level body cap.

A malicious Ollama substitute can stream a multi-GB response (or a deeply-nested JSON document) into memory. The response-truncation helper (`truncate(&body, 500)`) is used only on the error path, so it does not defend this case.

## Reproduction

1. Malicious `127.0.0.1:11434` responds `200 OK` with `Content-Length: 5000000000` and streams `{"models":[` + `{"name":"x"},`-forever.
2. Dashboard refresh stalls until OOM.

## Remediation

Bound the response before deserializing:

```rust
let bytes = resp.bytes().await?;
if bytes.len() > 1_048_576 {
    return Err(...);
}
let value: serde_json::Value = serde_json::from_slice(&bytes)?;
```

Also apply a short `.timeout()` on the client for this request (ties into H5).
