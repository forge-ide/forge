## Scope

Area: `crates/forge-shell`. Threat class: **T1 ŌĆö tool-approval plumbing**. The four-scope approval UI's security rests on the user reviewing the call in Session A's window. Approval is conveyed over `invoke('session_approve_tool', ...)`, but any window can issue that `invoke` ŌĆö so the decision channel is wider than the review channel.

## Finding

- **Location:** `crates/forge-shell/src/ipc.rs:43-111`, `crates/forge-shell/src/bridge.rs:50-71`
- **Severity:** high
- **Threat class:** T1

Every `#[tauri::command]` handler takes `session_id: String` from the caller and resolves it against a single app-wide `SessionConnections` HashMap on the `AppHandle`. The handler does not check `AppHandle::get_webview_window(...)` to confirm the calling window's label matches `session-{session_id}` (or `dashboard` for listing commands). The Tauri capability file gives identical `core:default` access to every `session-*` window and the dashboard.

Session IDs are not secrets ŌĆö `session_list` returns them to the dashboard, and they're written to `meta.toml` on disk. Consequences:
- If any Session-B window is compromised or buggy, it can call `session_approve_tool(session_A_id, toolCallId, 'ThisTool')` and bless a destructive tool call in Session A without the user ever seeing the prompt there.
- A single injection in the dashboard (which inherently holds every live session ID) can do this for all sessions simultaneously.

This interacts multiplicatively with the CSP-null finding: any XSS in any window = approval control over every session.

## Reproduction

1. Open Session A, wait for a tool-call-pending event. Copy `tool_call_id` from Session A's window.
2. In the dashboard (or any other window's devtools): `await window.__TAURI_INTERNALS__.invoke('session_approve_tool', { sessionId: 'A-id', toolCallId: 'capturedId', scope: 'ThisTool' })`.
3. The daemon for Session A receives `ToolCallApproved` and runs the tool ŌĆö Session A's UI never prompted.

See `crates/forge-shell/tests/ipc_commands.rs:80-96` for exact `invoke` shape.

## Remediation

In each session command handler, take the `webview::Webview` argument (Tauri injects it) and assert:

```rust
if webview.label() != format!("session-{}", session_id) {
    return Err(tauri::Error::from(anyhow!("forbidden: window label mismatch")));
}
```

For listing/opening commands, assert `webview.label() == "dashboard"`. Add a regression test using `mock_builder` with two named windows asserting cross-window invokes are rejected.
