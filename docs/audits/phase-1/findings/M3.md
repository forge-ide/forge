## Scope

Area: `crates/forge-fs`. Threat class: **T10 — panic/DoS on untrusted input**. `forge-fs` is the substrate under `fs.read/write/edit`, all of which accept LLM-supplied paths. Once `allowed_paths` is tightened (H7), it may still admit directories containing very large files (`/var/log/**`, workspace artifacts, release binaries). A single oversized tool call then OOMs the daemon.

## Finding

- **Location:** `crates/forge-fs/src/lib.rs:26` (`read_file`), `crates/forge-fs/src/mutate.rs:48-79` (`write`), `:83-106` (`edit`)
- **Severity:** medium
- **Threat class:** T10

`read_file` calls `std::fs::read(&canonical)` and then `String::from_utf8_lossy` over the full buffer — the whole file materializes in RAM. No cap before or during the read.

If `allowed_paths` includes any directory with a multi-GB file (logs, binaries, datasets, bundler caches), an LLM-issued `fs.read` pins hundreds of MB or more — enough to OOM `forge-session` on a resource-constrained machine.

Analogously, `write` and `edit` impose no cap on `content` / patch-derived output; multi-GB writes are limited only by disk.

## Reproduction

1. `allowed_paths` includes a directory with a large file.
2. Agent calls `fs.read` on it.
3. `read_file` allocates `Vec<u8>` of that size; `from_utf8_lossy` may allocate again.
4. On an 8-16 GB laptop, this is a single-request DoS.

`fs.write` with a 1 GB content string completes without objection.

## Remediation

Impose a configurable max-bytes cap (defaults: ~10 MiB read, ~10 MiB write/edit patch output), tunable via `ToolCtx` or a `forge-fs::Limits` struct.

For read, `stat` the file first via `std::fs::metadata(&canonical)` and reject *before* allocating.

For write/edit, reject if `content.len()` / final post-patch size exceeds the cap.

Keep the cap in `forge-fs` so enforcement is co-located with I/O, not relied on `forge-session`.
