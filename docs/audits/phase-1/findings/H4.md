## Scope

Area: `crates/forge-providers`. Threat class: **T8 — NDJSON stream-parsing DoS**. The `OllamaProvider` reads streaming NDJSON from `http://127.0.0.1:11434`. Any local process that binds that port before Ollama (race at startup, or after an Ollama crash) controls the byte stream. An unbounded buffer on the reader side is a single-request DoS.

## Finding

- **Location:** `crates/forge-providers/src/ollama.rs:100-144` (`futures::stream::unfold` over `bytes_stream`)
- **Severity:** high
- **Threat class:** T8

The streaming decoder accumulates bytes via `buf.push_str(&String::from_utf8_lossy(&b))` and only drains on newline. There is no maximum line length. A malicious peer can send an arbitrarily large payload with no newline; the decoder keeps waiting for a newline that never arrives, growing `buf` without bound. `from_utf8_lossy` additionally replaces invalid UTF-8 with U+FFFD (3 bytes each), so 1-byte non-UTF-8 sequences inflate memory 3x per byte.

## Reproduction

1. Bind `127.0.0.1:11434` with a script that accepts `POST /api/chat`, returns `HTTP/1.1 200 OK` with `Transfer-Encoding: chunked`, then streams endless bytes with no newline.
2. Launch `forged` with `--provider ollama:any-model` and send any prompt.
3. Observe RSS of the `forged` process grow unbounded until OOM-kill.

## Remediation

Cap per-line reads before accumulating. Preferred: switch to a bytes decoder (`bytes::BytesMut`) with `tokio_util::codec::LinesCodec::new_with_max_length(1 << 20)` (1 MiB), which returns `LinesCodecError::MaxLineLengthExceeded` cleanly. Minimal: inline check — when `buf.len()` would exceed 1 MiB, return an error chunk from the stream and close the connection.

Also cap cumulative unflushed bytes to prevent slow-leak variants.
