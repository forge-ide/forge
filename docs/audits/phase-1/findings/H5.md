## Scope

Area: `crates/forge-providers`. Threat class: **T8 — resource exhaustion**. A session daemon that wedges on a single provider request is unavailable for every other operation (approvals, archive-on-SIGTERM). Timeouts are the standard defense against slow-drip and half-open attacks; their absence is a wedging primitive for any malicious peer that controls the Ollama port.

## Finding

- **Location:** `crates/forge-providers/src/ollama.rs:22` (`client: reqwest::Client::new()`)
- **Severity:** high
- **Threat class:** T8 (and T7 — any long-held connection blocks the daemon)

`OllamaProvider::new` constructs `reqwest::Client::new()` with no timeouts configured. A malicious `127.0.0.1:11434` squatter can accept the TCP connect, then stall forever during body read, or dribble one byte per minute to keep the stream alive indefinitely.

The only termination comes from the peer or the OS, not from Forge. Both `/api/chat` (streaming) and `/api/tags` (buffered JSON) paths are affected. A single malicious request can wedge a session's provider task; `list_models()` on dashboard refresh is equally affected.

## Reproduction

1. Start a `netcat` listener on `127.0.0.1:11434` that accepts the TCP connection and never writes.
2. Run `forged` with `--provider ollama:<model>` and send a prompt.
3. The client awaits `send()` / `bytes_stream().next()` indefinitely — no timer fires.

## Remediation

Replace `reqwest::Client::new()` with:

```rust
reqwest::Client::builder()
    .connect_timeout(Duration::from_secs(5))
    .read_timeout(Duration::from_secs(30))
    .timeout(Duration::from_secs(120))
    .tcp_keepalive(Some(Duration::from_secs(30)))
    .build()?
```

For the streaming `/api/chat` path, a total `.timeout()` would cut long generations — use per-read `.read_timeout()` instead (supported since reqwest 0.12.5). For `list_models()` (short-lived), apply a short total timeout.
