## Scope

Area: `crates/forge-core`. Threat class: **T1 (forward-looking)**. `SessionMeta` is the session-identity record on disk and the natural place to grow trust-bearing fields (allowed_paths, sandbox_profile, credential_ref). Without `deny_unknown_fields`, an older-version daemon silently ignores trust-relevant fields a newer daemon wrote.

## Finding

- **Location:** `crates/forge-core/src/meta.rs:8-29`, also `crates/forge-core/src/workspaces.rs:7-19`
- **Severity:** low
- **Threat class:** T1 (forward-looking)

`SessionMeta` derives `Deserialize` without `#[serde(deny_unknown_fields)]`. Today the struct is benign — F-038 intentionally moved provider selection to CLI flag / `FORGE_PROVIDER` env rather than meta. But rollback after a newer daemon writes new trust-relevant fields is indistinguishable from empty values.

## Reproduction

```bash
echo 'allowed_paths = ["/etc"]' >> <session-dir>/meta.toml
# read_meta returns Ok(SessionMeta { ... }) silently, field dropped
```

Future `SessionMeta` adding `allowed_paths: Vec<PathBuf>` with `#[serde(default)]` — rolling back after a newer daemon writes such a meta is equivalent to `allowed_paths = []`.

## Remediation

- Add `#[serde(deny_unknown_fields)]` to `SessionMeta`
- Couple with `#[serde(default = "schema_v1")] pub schema: u32` so migrations are intentional
- Same treatment for `WorkspaceEntry` (`workspaces.rs:7`) and `WorkspacesFile` (`:15`)
