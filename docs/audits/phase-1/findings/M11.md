## Scope

Area: `crates/forge-shell`. Threat class: **T5 — capability scope**. The capability file's glob + unsanitized window label creates brittle assumptions that will break silently on a future Tauri upgrade or capability refactor.

## Finding

- **Location:** `crates/forge-shell/src-tauri/capabilities/default.json:5`, `crates/forge-shell/src/window_manager.rs:33`, `crates/forge-shell/src/window_spec.rs:49`
- **Severity:** medium
- **Threat class:** T5

The capability's `windows` array is `["dashboard", "session-*"]`. Tauri resolves `session-*` as a glob over the window label. `open_session(id)` constructs the label as `session-{id}` from an attacker-controllable string (Tauri command signature is `id: String`).

A caller passing `id = "../../something"` or an id with NUL / whitespace yields labels like `session-../../something`, which still match `session-*`.

Today every window has the same permission set so there is no immediate confusion. But as soon as the capability file splits permissions (dashboard gets narrower access), the glob + unsanitized id means any Session window can be given a label starting with `session-` but containing unexpected characters.

Additionally, the file's comment asserts *"App-local commands do not require capability entries — Tauri only gates plugin commands"*. This is true for Tauri 2 today but is an unstable assumption. A future Tauri that gates app-local commands will silently open everything.

## Reproduction

1. From the dashboard, invoke `open_session` with `id = "../something"`.
2. Observe a window with label `session-../something` matches the capability.

## Remediation

1. Validate `id` in `dashboard_sessions::open_session` before building the label: reject any id not matching `^[a-f0-9-]{1,64}$` (canonical `SessionId`).
2. Add explicit capability entries listing the seven app-local commands even though Tauri does not currently require them, so a Tauri upgrade cannot silently widen or narrow the attack surface without a capability-file diff.
